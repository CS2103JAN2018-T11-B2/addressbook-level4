PROJECT: PrisonBook

Overview

Prisonbook is a desktop address book application used for Prison Guards/Inmates management. The user interacts with it using a CLI, and it has a GUI created with JavaFX. It is written in Java, and has about 10 kLoC.

Adding a prisoner to a cell: addcell or ac (v1.3)

Adds a person to one of the cells in the prison
Format: ac INDEX CELLADDRESS

Person added must be a prisoner and not already in a cell within the prison. The cell must be in the map and not full.

Examples:

  • addcell 1 1-1

  • ac 2 2-1 === Listing prisoners in a cell : listcell or lc (coming in V2.0)

Shows a list of all persons in a prison cell in the prison book.
Format: list CELL

Examples: * listcell a1 * lc c13 === Showing cells in the map: map or m (v1.1)

Shows a table of prison cells like a map.
Format: map === Deleting a person from a cell : deletecell or dc (coming in V2.0)

Deletes the specified person from the cell in the PrisonBook.
Format: deletecell INDEX

  • Deletes the person at the specified INDEX.

  • The index refers to the index number shown in the most recent listing.

  • The index must be a positive integer 1, 2, 3, …​

Examples:

  • lc a2
    deletecell 2
    Deletes the 2nd person in the cell in the PrisonBook.

  • listcell b4
    dc 1 Deletes the 1st person in the cell in the PrisonBook.

Prison Cell Map

Current Implementation

In a prison, it is necessary to have cells, where prisoners stay in. It is important to keep track of the prisoners in the cell and whether the Cell is full.

This cell system is most clearly represented using a 2-D array. Hence, this was used as the basis of the map system of the prison. When implemented, it will take up the right side area of the application (but this has not yet been implemented). Below is an idea of what the UI may look like by the end of the project. The current implementation is still shown in the box below the user input. This basic map shows the basic implementation of the cell system and helps keep track of what is happening in the cells.

With this map system of cells, the cells can be viewed. Not only that, prisoners can also be added and removed from them. These changes can be viewed in the bigger map which displays the number of people in them (also shown in the UI mockup above, represented by the square brackets) or in individual cells.

For future implementations, we can also allow only the prison wardens/owners to redefine the maximum size of the prison as their prisons may grow or shrink in the future. In this scenario, we will strive only to hit our non-functional requirements.

Assuming the user has access to the functions to add prisoners into prison cells, they can use the addcell command to do so. Below is the sequence diagram.

The cell map will have all the cells and its current occupants.

Besides the cell map system being updated, we will also update the addresses of these prisoners added to the prison, since they now live in the prison, rather than in their own homes.

For future implementations, there will also be a command to remove prisoners, when they are being transferred to another cell, or if they can leave prison and go back home to their loved ones. This will be done in a similar manner to addcell.

In much further implementations, guards/prison workers can also be assigned to cells, not as prisoners but as according to their job descriptions.

Design Considerations

Aspect: Data Structure of Cells in CellMap

  • Alternative 1 (current choice): Use a 2-D array`

    • Pros: Easy to visualise the system.

    • Cons: Harder to do more functions that the Java API has provided. Does not fit in with the other models of Person and Tag with their UniquePersonList and UniqueTagList

  • Alternative 2: Use a 2-level ArrayList`

    • Pros: Can use Java functions meant for Lists, can have different number of units for each block

    • Cons: Not uniform number of units for each block will be harder to take into consideration when using the individual cells.

Aspect: Implementation of ShowCellsCommand (map)

  • Alternative 1 (current choice): Create a new Command to show the map.

    • Pros: Easy to implement, similar to listing. Extending the current list of commands. Only when function is called can the map be seen.

    • Cons: Easy to forget which cell in the map can add prisoners but the map is not permanently there.

  • Alternative 2: Keep the map always on display.

    • Pros: Easier to see when adding prisoners to cells.

    • Cons: Risk security as access to map is not limited. Even if it is, anyone walking by can see the information easily.

Aspect: Implementation of AddCellCommand (addcell)

  • Alternative 1 (current choice): Prisoners are added into cells in a cellMap.

    • Pros: Cells are always there and can hold prisoners. Since these groups are still Persons with a guard role or prisoner role, not all persons may have a cell.

    • Cons: More difficult to see the individual’s cell. (have not implemented updating addresses)

  • Alternative 2: Not have a cell map but to add cells to prisoners.

    • Pros: Might be more intuitive for the user, to check a prisoner’s cell location.

    • Cons: Harder to manage a system of cells. Easy to accidentally overcrowd a cell.

  • Alternative 3: Automatically assigning cells to prisoners

    • Pros: Removes the burden of choosing a cell for user. Fewer commands required as prisoners are automatically added.

    • Cons: More complex in automatically choosing cells for prisoners. User might want to have control on which prisoners should be in which cell.

Appendix B: Use Cases

Use case: Add a prisoner

Actor: Guard (Guard)

MSS

  1. Guard requests to add prisoner

  2. PrisonBook adds prisoner

  3. Guard requests to list cells that are empty/not filled

  4. PrisonBook shows a list of empty/not filled cells

  5. Guard requests to view a specific cell’s prisoner’s details

  6. PrisonBook shows details of prisoner in cell

  7. Guard requests to add a prisoner to a specific cell

  8. PrisonBook adds prisoner to cell

    Use case ends.

Extensions

  • 2a. There are no cells available for new prisoner.

    • 2a1. PrisonBook shows an error message.

      Use case ends.

  • 5a. The given index is invalid, outside range.

    • 5a1. PrisonBook shows an error message.

      Use case resumes at step 4.

  • 5b. The given index of cell has no prisoners.

    • 5b1. PrisonBook shows an error message.

      Use case resumes at step 4.

  • 7a. The given index is invalid.

    • 7a1. PrisonBook shows an error message.

      Use case resumes at step 6.

Use case: Assign prisoner to a cell

Actor: Guard

MSS

  1. Guard selects a prisoner

  2. Guard requests a list of available cells with room available

  3. PrisonBook shows a list of available cells

  4. Guard requests to add prisoner to vacant cell

  5. PrisonBook updates prisoner status and cell status

    Use case ends.

Extensions

  • 2a. The cell list is empty.

    Use case ends.

  • 3a. Prisoner is already assigned to a cell.

    • 3a1. PrisonBook removes cell tag from prisoner status and removes prisoner from cell status

      Use case resumes at step 2.

Use case: Get the list of prisoners in a certain cell

Actor: Guard

MSS

  1. Guard requests a list of cells

  2. PrisonBook shows the list of cells

  3. Guard requests to view the list of prisoners assigned to a specific cell

  4. PrisonBook shows a list of of prisoners assigned to the cell

    Use case ends.

Extensions

  • 3a. Guard does not have access rights to view the details of the requested cell

    Use case ends.
